## 版本說明與核心結構

### 版本說明與主要更新摘要

本版的核心目標，不再是補充更多威脅名詞，而是把 v0.2 從「結構化字典」升級為「可以被風險評估工具、案例比對引擎、規則引擎、AI 助理與安防運營平台直接調用」的知識底座。設計原則仍保留 ATT&CK 式的「威脅主體—資產—戰術—技術—路徑—脆弱性—措施—案例」骨架，但進一步把**指標、能力、證據、控制效果與行業畫像**獨立成可計算對象。這一方向與 NIST 對控制與 assurance 的結合邏輯、NPSA 對 hostile reconnaissance、insider risk、UAS、data centre security 與 CAPSS 的整體防護思路，以及 FBI/OSHA/DOE 對主動暴力、醫療場域、關鍵能源基礎設施的實務導向一致。citeturn12view0turn31view0turn26view0turn28view0turn28view2turn32view3turn18view3turn12view3turn22view2

本次補齊的五大缺口如下。第一，新增**行為指標庫 Indicators**，把原本散落在技術條目中的觀測線索抽離成標準化、可編碼、可檢索的指標；第二，新增**威脅主體能力模型 Capabilities Model**，使「誰在做」能夠被量化，而不是僅靠主觀描述；第三，新增**技術—控制效果映射矩陣**，讓技術、控制、證據、殘餘風險之間形成可驗證關聯；第四，新增**證據與置信度模型**，使案例映射、告警判讀與 AI 推理不再只靠單一事件；第五，新增**行業畫像庫**，讓同一套知識庫可以落地到校園、醫療、製造、資料中心、零售、能源與政務等高頻場景。這種做法也更貼近近年國際保護性安全實踐對「全威脅、全災害、分層防護、持續運營」的要求。citeturn8news2turn4view4turn30view0turn22view2

### 七 新增 威脅主體能力模型

能力模型採 1–5 分制，分數越高代表該能力越強。為方便工具計算，本版使用八維向量：  
`O 組織化、R 資源能力、I 情報能力、T 技術能力、V 暴力能力、A 接近權、P 持續性、E 規避能力`

**分數定義**

| 分數 | 含義 |
|---|---|
| 1 | 極低，偶發或幾乎不具備 |
| 2 | 低，僅能依賴簡單手法 |
| 3 | 中，可在熟悉場景重複使用 |
| 4 | 高，具備計畫、分工或專業支援 |
| 5 | 極高，可跨場景、長期、複合式運作 |

**建議權重**

- 以內部濫用風險為主：A、P、E 權重較高  
- 以暴力事件風險為主：V、I、P 權重較高  
- 以關鍵基礎設施風險為主：I、T、E、P 權重較高  
- 以零售 / 倉配失竊為主：O、R、E、A 權重較高

| 等級 | 名稱 | 判定規則 |
|---|---|---|
| C0 | 未知 | 無可用證據 |
| C1 | 低 | 單一來源、且為易誤判來源 |
| C2 | 中 | 兩種來源，或單一高品質系統證據 |
| C3 | 中高 | 至少兩種獨立來源，其中一種為系統 / 視訊 / 勘驗證據 |
| C4 | 高 | 三種以上來源互相印證，且包含至少一種高完整性證據 |

**證據融合規則**

```text
1. 單一訪談或單一告警，不得直接映射為高置信技術判定。
2. 同一設備產生的多筆紀錄，只算一個獨立來源。
3. 若存在視訊 + 門禁 + 工單/預約三者一致，通常可升至 C4。
4. 若存在管理缺陷但無事件證據，可判為脆弱性成立，不可直接判為攻擊已發生。
5. 對 AI 推理引擎，必須保留 evidence_trace、confidence_score、contradictions 欄位。
```

### 十五 主要更新點總結與後續擴展建議

本次 v0.3 的實質升級，可以概括為五點。第一，**把技術變成可觀測的指標**，因此可以支援安防營運中心、PSIM、門禁審計與 AI 規則引擎。第二，**把主體變成可量化的能力向量**，因此可以在不同行業與場景做威脅成功可能性排序。第三，**把控制從設備清單變成效果矩陣**，因此可以回答「哪些控制對哪種技術有效、需要什麼證據、還剩多少風險」。第四，**把案例從敘事升級為結構化映射**，可直接支援相似案例比對與覆盤。第五，**把通用知識庫加上行業畫像層**，避免重新落回傳統 checklist。這些方向都與近年官方資料所強調的分層保護、全威脅韌性、安防系統網路化風險、主動暴力預防與關鍵基礎設施韌性相一致。citeturn12view0turn31view0turn4view4turn26view0turn18view3turn22view2

後續擴展建議有四項。其一，將 `IND` 與你現有日誌平台、門禁、VMS、工單系統正式對接，形成 `indicator_event` 事件表。其二，建立 `path_scoring` 與 `control_coverage` 計算器，讓風險評估從人工主觀打分走向可重複。其三，為每個行業建立更細的子模板，例如校園拆分 K-12 / 大專、醫療拆分急診 / 住院 / 藥局、能源拆分變電站 / 調度中心 / 油氣站。其四，逐步把 `CASE` 與 `IND` 做反向學習，讓 AI 可以根據案例自動推薦最可能的技術鏈與最弱控制。這一步與 FBI / NPSA 近年的「從前兆到干預」思路，以及 NIST 對 assurance 與持續監控的思路是一致的。citeturn20view3turn28view0turn12view0
